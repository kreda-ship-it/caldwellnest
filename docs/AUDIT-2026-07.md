# CaldwellNest Code Audit — July 2026

A full read-through of `index.html` (all ~8,800 lines), done 2026-07-11.

Line numbers refer to the single-file version at commit `0974486`. If the JavaScript later moves
into separate files, search for the **function name** instead of the line number.

**How to read severity**
- 🔴 **High** — real risk to users or data. Fix before inviting more students.
- 🟠 **Medium** — wrong behavior in situations students will actually hit. Fix soon, one at a time.
- 🟡 **Low** — cleanup, dead code, or "this will bite us later." Batch these when already working in that area.

Nothing in this document has been fixed except the table directly below. Each item is a candidate
for its own small, focused change — pick one, fix it, test it, commit it.

## 🔴 High

### H1. Stored XSS — student-typed text is injected into the page as HTML — ✅ FIXED 2026-07-13
> **Status: fixed.** `esc()` and `escAttr()` live in `js/utils.js` and are applied at every user-text
> interpolation site (213 of them). `tests/xss-test.html` renders a listing whose title, description,
> location, and poster name are live payloads and asserts nothing executes — open it over a local
> server (`python3 -m http.server`) and it prints PASS/FAIL. Re-run it after touching any card,
> bubble, or admin-table template. The original write-up is kept below for the reasoning.
**Where:** everywhere the app builds HTML with `innerHTML` and drops user text straight in.
Confirmed spots include `listingCardHTML()` (title, location, description), `posterHeaderHTML()` (name),
`quoteHtml()` and `startReply()` (message text), `openConvo()` chat bubbles, `renderConvos()` (names + previews),
`viewStudentProfile()` (display name, bio, pronouns), the admin approval cards, `openListingDrawer()`,
`renderAReports()` (report details), and `_appealCard()` (appeal message).

**What happens:** a student posts a listing titled `<img src=x onerror="...">`. That is not shown as text —
the browser *runs* it, in the browser of every student who sees the card, and in the admin's browser when it
reaches the moderation queue. Because the admin is logged in with more permissions, the admin's session is the
real prize here.

**Why it exists:** the codebase has an `escAttr()` helper (used 6 times, and only for quoted attributes like
`alt="..."`), but there is no equivalent for text placed *between* tags — which is most of the app.

**The fix (one focused change):** add one `esc()` helper next to `escAttr()`:
```js
function esc(s) {
  return String(s ?? '').replace(/&/g,'&amp;').replace(/</g,'&lt;')
                        .replace(/>/g,'&gt;').replace(/"/g,'&quot;').replace(/'/g,'&#39;');
}
```
then wrap every user-supplied value at the point it enters a template string: `${esc(l.title)}`,
`${esc(m.content)}`, `${esc(p.name)}`, and so on. Do **not** wrap values the code itself controls
(category labels, emoji constants, computed colors) — only things a person typed.

This is a big sweep, so do it as its own commit and nothing else. Good news: the NestBot markdown path
already has a real sanitizer (`sanitizeMd()`), so that surface is fine.

### H2. Anyone can log in as the demo student
**Where:** `DEMO_EMAIL` / `DEMO_PASSWORD`, near the demo-login function.
The demo account's real password sits in plain text in a file the browser downloads. Anyone who opens
"view source" can sign in as that account and post, message students, and file reports as a "verified" student.

**Decide, don't just patch:** for a small pilot this may be an acceptable trade for a one-click demo. But it
should be a *decision*, not an accident. Options: keep it but strip the demo account's ability to post/message;
or move the demo behind a short-lived server-issued token later, when a backend exists.

## 🟠 Medium

### M1. Books and listings disagree about "pending sale" — ✅ FIXED 2026-07-13
> **Status: fixed.** Books now follow the housing rule exactly: **pending sale means people are interested,
> so the item stays in the feed with a badge** — it tells other buyers to hurry or move on. `loadBooks()` now
> fetches `active` *and* `pending_sale`, and `isListingLive()` is the single gate that decides visibility.
> Book owners got the matching actions (Mark pending sale / Back to active) via a new `bookLifecycleAction()`
> — deliberately **not** the marketplace `lifecycleAction()`, which looks ids up in `DB.listings`; book ids and
> listing ids are independent sequences, so reusing it would act on an unrelated listing. Buyers now see the
> badge on the card *and* a note in the book detail before they message. Original write-up below.

`isListingLive()` — the one rule the feed uses — treats a `pending_sale` item as **visible** (deliberately:
buyers should see a deal is in progress). But `loadBooks()` only fetched books with `lifecycle_status = 'active'`.

**Result:** a book marked pending-sale silently vanished from the browse feed on the next reload, while a room
listing in the same state stayed visible with a badge. Worse, that book was still visible on the seller's public
profile (`viewStudentProfile()` doesn't filter by lifecycle), so the same book was present in one place and gone
from another.

### M2. The edition filter is built from the live books cache — ✅ RESOLVED BY M1
`buildCatFiltersHTML()` builds the Edition dropdown from `_books`. Now that `_books` holds exactly what the feed
shows (active + pending_sale), the dropdown matches the feed by construction. No longer a trap.

### M3. `loadListings()` gives up quietly when the table is empty — ✅ FIXED 2026-07-13
> **Status: fixed.** A *failed* query and an *empty* table are now handled differently: on failure the cached
> listings are kept (stale beats a blank feed); on a genuinely empty result the caches are assigned `[]`.

`if (data.length === 0) return;` returned *before* assigning `DB.listings`, so the previous (stale) list stayed
on screen. Deleting the last listing, or an empty database, still showed a populated feed.

### M4. Admin dashboard does a full reload every time anyone sends a chat message — ✅ FIXED 2026-07-13
> **Status: fixed.** `messages` is off the full-reload path. It now has its own channel and handler
> (`scheduleMessageCountRefresh`) that runs a single debounced HEAD count query to update the message stat, and
> re-renders the Messages table only when that section is actually on screen. The other five tables (listings,
> book_listings, reports, appeals, profiles) still trigger the full `adminReload()`, which is correct — those
> genuinely change the queues, counts and charts. Teardown was also centralised into
> `stopAdminRealtimeListeners()`, which logout now calls.

Every message any student sent to any other student used to trigger a debounced full admin reload — re-fetching
every listing, every book, every poster profile, all badge counts, school stats, charts and the open section,
all so one number could tick up.

### M5. Search crashes if a listing has no title — ✅ FIXED 2026-07-13
> **Status: fixed.** Guarded in both places: the feed's `kwMatch` (`js/listings.js`) and the chat listing
> picker's filter (`js/messages.js`) — the picker had the identical unguarded call.

`l.title.toLowerCase()` with no guard: a row with a null title made typing in the search box throw, and the feed
stopped rendering.

### M6. The school list is cached for the whole session
`loadSchools()` returns early if `_schoolsList` is non-empty, so a school added while a student has the tab open
never appears until they reload. Minor, but it silently breaks the distance filter for the new school.

## 🟡 Low

- **L1. `ago()` was defined twice — ✅ FIXED 2026-07-13.** Two `function ago(...)` declarations existed. Because
  JavaScript hoists function declarations, **the last one wins for the whole scope** — the first was unreachable
  dead code, and `const _ago_orig = ago` captured the *second* one, not the original it appeared to save. Both
  the dead definition and the misleading `_ago_orig` are gone; the surviving `ago()` (next to `_agoMap`) was
  already a strict superset. Verified by routing all 16 admin sections through it.
- **L2. Legacy in-memory stores — ⚠️ half fixed.** `DB.convos`, `DB.reports`, and `DB.students` are declared but
  every real surface queries Supabase directly.
  - ✅ The *actively wrong* part is fixed: NestBot's fallback counted `DB.reports` (never populated) and so
    confidently answered **"0 open reports"** even when reports were waiting. It now asks the database.
  - ⬜ Still open: the empty `DB.convos` / `DB.reports` / `DB.students` arrays themselves. **Deleting them needs
    Kal's OK** — it means removing code, and `DB.students` is still referenced by the dead verify screen (L3).
- **L3. `PENDING_VERIFY` is permanently empty — ⬜ needs a decision, not a fix.** The whole Student Verification
  admin section renders from an array that is never filled (Supabase handles .edu gating). It is a dead screen
  showing "0 pending" forever. Removing it means deleting an admin section + its HTML — **not done without
  Kal's explicit OK**, per the "never delete large sections" rule in CLAUDE.md.
- **L4. In-memory admin state resets on refresh.** `DB.log` (activity feed) and `DB.content` (site editor) live
  only in memory. Already known and documented in CLAUDE.md — listed here so the audit is complete.
- **L5. No `.gitignore` — ✅ FIXED 2026-07-13.** `.DS_Store` is ignored.
- **L6. Inline styles in generated HTML — 🔄 IN PROGRESS.** There is no `<style>` block (good — `styles.css` is
  the only stylesheet), but there were **1,084** inline `style="..."` attributes across the JS templates and
  `index.html`.

  **Done (2026-07-13) — the structural part, which is where the pain actually was.** The problem was never the
  raw count; it was that helper functions *emitting* styled markup had been copy-pasted: `secHead` ×3,
  `chipRow` ×3, `sPill` ×4 — ten definitions of three things, each carrying its own inline styles. Those are now
  three shared helpers (`aFieldLabel`, `aChipRow`, `aStatusPill`) backed by real classes
  (`.a-field-label`, `.a-chip-row`, `.pill-compact`), plus `.df-label` for a label string that appeared 11 times
  in `listings.js`. Each new class was checked to declare *exactly* what the inline style it replaced declared,
  so nothing moved a pixel.

  **Phase 2 (2026-07-13) — `index.html`.** 100 more inline styles became 16 semantic classes
  (`.label-hint`, `.exp-card`/`.exp-body`/`.exp-desc`/`.exp-btn`, `.hiw-card-title`/`.hiw-card-text`,
  `.check-label`, `.modal-narrow`, `.table-scroll`, `.link-brand`, `.bg-alt`/`.bg-surface`, …).
  **472 → 372** in `index.html`; **1,084 → 965** overall.

  Two traps this phase surfaced — both now guarded, and both worth knowing before continuing:
  - **The 28 inline `display:none` attributes are load-bearing. Do not move them.** The JS shows elements with
    `el.style.display = ''`, which falls back to the CSS. If `display:none` lived in a class, that reset would
    *re-hide* the element instead of showing it. They stay inline on purpose.
  - **Check for class-name collisions before appending a rule.** The first attempt added a `.form-label` rule —
    a name that already existed at line 33 — which silently resized *every* form label in the app. Later rules
    win. `tests/style-snapshot.js` caught it; the conversion script now aborts on a name clash.

  **How to do the rest safely:** `tests/style-snapshot.js` records the *computed* style of all 1,543 elements
  (hidden modals and the whole admin panel included) so you can prove a refactor changed nothing. Both phases
  above were verified this way — 106,467 computed values, byte-identical before and after.

  **Phase 3 (2026-07-14) — category colours.** 21 more (`.tcard-flush`, `.note-xs`, and the
  `.cat-*` / `.btype-*` picker buttons). **351 in `index.html`; ~944 overall.**

  ⚠️ **Two things phase 3 turned up that are NOT styling issues — worth a decision:**
  1. **The category colours are duplicated.** The post-category picker buttons hardcoded the exact
     hex values from `CATEGORY_COLORS` in `js/config.js`. They are now `.cat-*` classes in
     `styles.css`, but that is still *two* copies of the palette: JS paints listing cards from the
     object, CSS paints the picker. Change one and the other silently drifts. Single-sourcing them
     (CSS custom properties both sides read) is a genuine follow-up.
  2. **The Books picker button is the wrong colour.** It is painted `#E8EEF5` — the *Technology*
     blue — while book cards in the feed render `#E6EFEF` (the muted teal in `CATEGORY_COLORS.books`).
     Almost certainly stale, from before books got their own colour. **Left as-is on purpose:**
     repainting it is a visual/brand change, not a refactor. Kal's call.

  **Remaining: ~944.** A genuine long tail (most common single pattern: 29 occurrences), in two piles:
  `admin.js` (**~460**, inside template literals) and `index.html` (**372**, static markup). Convert things that
  are semantically a *component*; leave true one-offs inline. Do **not** invent a pile of one-off utility
  classes (`.u-text-center` etc.) — that just moves the mess.
