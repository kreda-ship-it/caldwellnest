# CaldwellNest Code Audit — July 2026

A full read-through of `index.html` (all ~8,800 lines), done 2026-07-11.

Line numbers refer to the single-file version at commit `0974486`. If the JavaScript later moves
into separate files, search for the **function name** instead of the line number.

**How to read severity**
- 🔴 **High** — real risk to users or data. Fix before inviting more students.
- 🟠 **Medium** — wrong behavior in situations students will actually hit. Fix soon, one at a time.
- 🟡 **Low** — cleanup, dead code, or "this will bite us later." Batch these when already working in that area.

Nothing in this document has been fixed except the table directly below. Each item is a candidate
for its own small, focused change — pick one, fix it, test it, commit it.

## Fixed during this session (for the record)

| Commit | What was broken |
|---|---|
| `01979cd` | **Listings appearing/disappearing on reload.** `initStudent()` started loading data at the same moment Supabase was restoring the saved login. When the data request won that race, it ran as an *anonymous* visitor — and the database's security rules (RLS) hide rows from anonymous visitors, so the feed came back partial or empty. Boot now waits for the session before loading anything. |
| `77cf6be` | **Listings changing while just sitting there (admin).** Two causes. (1) The admin auto-refresh could start a second reload before the first finished; whichever response landed last won, so an *older* answer could overwrite a newer one. Reloads are now serialized. (2) If the "who is suspended?" lookup failed, the code treated it as "nobody is suspended" and un-hid those listings for a cycle. It now reuses the last good answer. |
| `0974486` | **Ghost refreshes after admin logout.** The six admin realtime listeners stayed subscribed after signing out and kept firing full reloads as an anonymous client. They are now shut down on logout. |

## 🔴 High

### H1. Stored XSS — student-typed text is injected into the page as HTML
**Where:** everywhere the app builds HTML with `innerHTML` and drops user text straight in.
Confirmed spots include `listingCardHTML()` (title, location, description), `posterHeaderHTML()` (name),
`quoteHtml()` and `startReply()` (message text), `openConvo()` chat bubbles, `renderConvos()` (names + previews),
`viewStudentProfile()` (display name, bio, pronouns), the admin approval cards, `openListingDrawer()`,
`renderAReports()` (report details), and `_appealCard()` (appeal message).

**What happens:** a student posts a listing titled `<img src=x onerror="...">`. That is not shown as text —
the browser *runs* it, in the browser of every student who sees the card, and in the admin's browser when it
reaches the moderation queue. Because the admin is logged in with more permissions, the admin's session is the
real prize here.

**Why it exists:** the codebase has an `escAttr()` helper (used 6 times, and only for quoted attributes like
`alt="..."`), but there is no equivalent for text placed *between* tags — which is most of the app.

**The fix (one focused change):** add one `esc()` helper next to `escAttr()`:
```js
function esc(s) {
  return String(s ?? '').replace(/&/g,'&amp;').replace(/</g,'&lt;')
                        .replace(/>/g,'&gt;').replace(/"/g,'&quot;').replace(/'/g,'&#39;');
}
```
then wrap every user-supplied value at the point it enters a template string: `${esc(l.title)}`,
`${esc(m.content)}`, `${esc(p.name)}`, and so on. Do **not** wrap values the code itself controls
(category labels, emoji constants, computed colors) — only things a person typed.

This is a big sweep, so do it as its own commit and nothing else. Good news: the NestBot markdown path
already has a real sanitizer (`sanitizeMd()`), so that surface is fine.

### H2. Anyone can log in as the demo student
**Where:** `DEMO_EMAIL` / `DEMO_PASSWORD`, near the demo-login function.
The demo account's real password sits in plain text in a file the browser downloads. Anyone who opens
"view source" can sign in as that account and post, message students, and file reports as a "verified" student.

**Decide, don't just patch:** for a small pilot this may be an acceptable trade for a one-click demo. But it
should be a *decision*, not an accident. Options: keep it but strip the demo account's ability to post/message;
or move the demo behind a short-lived server-issued token later, when a backend exists.

## 🟠 Medium

### M1. Books and listings disagree about "pending sale"
`isListingLive()` — the one rule the feed uses — treats a `pending_sale` item as **visible** (deliberately:
buyers should see a deal is in progress). But `loadBooks()` only fetches books with `lifecycle_status = 'active'`.

**Result:** a book marked pending-sale silently vanishes from the browse feed on the next reload, while a room
listing in the same state stays visible with a badge. Worse, that book is still visible on the seller's public
profile (`viewStudentProfile()` doesn't filter by lifecycle), so the same book is present in one place and gone
from another.

**Fix:** make `loadBooks()` fetch `active` *and* `pending_sale`, and let `isListingLive()` be the only thing
that decides visibility — the same way marketplace listings already work.

### M2. A book marked "sold" hides the ISBN/edition filter options, not just the book
`buildCatFiltersHTML()` builds the Edition dropdown from `_books` — the *live* books cache. This is correct
today, but it means the filter list silently changes shape as books sell. Not a bug yet; noted because it is a
trap when M1 is fixed.

### M3. `loadListings()` gives up quietly when the table is empty
`if (data.length === 0) return;` — it returns *before* assigning `DB.listings`, so the previous (stale) list
stays on screen. On a truly empty database the feed keeps showing whatever it had. The genuinely-empty case and
the error case should be told apart: on error, keep the cache (already fixed); on a real empty result, clear it.

### M4. Admin dashboard does a full reload every time anyone sends a chat message
`startAdminRealtimeListeners()` subscribes to the `messages` table with `event: '*'`. Every message any student
sends to any other student triggers a debounced full admin reload — listings, books, badges, charts, the open
section. It works, but it is a lot of database traffic for something the admin doesn't even see, and it is the
kind of thing that gets expensive with real usage.

**Fix:** either drop `messages` from the watched list, or (if the message counter matters) have it refresh only
the badge instead of the whole dashboard.

### M5. Search crashes if a listing has no title
`renderListings()` → `kwMatch` calls `l.title.toLowerCase()` with no guard. If a row ever lands with a null
title, typing in the search box throws and the feed stops rendering. Cheap insurance: `(l.title || '')`.

### M6. The school list is cached for the whole session
`loadSchools()` returns early if `_schoolsList` is non-empty, so a school added while a student has the tab open
never appears until they reload. Minor, but it silently breaks the distance filter for the new school.

## 🟡 Low

- **L1. `ago()` is defined twice.** There are two `function ago(...)` declarations (the second one near
  `_agoMap`, which handles more sections). Because JavaScript hoists function declarations, **the second one
  always wins** — the first is dead code, and `const _ago_orig = ago` does not capture what it looks like it
  captures. Delete the first definition and the unused `_ago_orig`.
- **L2. Legacy in-memory stores.** `DB.convos`, `DB.reports`, and `DB.students` are declared and pushed to in a
  few places, but every real surface now queries Supabase directly. The NestBot fallback still reads
  `DB.reports` and will always report 0 open reports. Either wire them up or delete them.
- **L3. `PENDING_VERIFY` is permanently empty.** The whole Student Verification admin section renders from an
  array that is never filled (Supabase handles .edu gating). It's a dead screen showing "0 pending" forever.
- **L4. In-memory admin state resets on refresh.** `DB.log` (activity feed) and `DB.content` (site editor) live
  only in memory. Already known and documented in CLAUDE.md — listed here so the audit is complete.
- **L5. No `.gitignore`.** `.DS_Store` is sitting untracked in the repo root.
- **L6. Inline styles everywhere in generated HTML.** The `<style>` block is gone (good — `styles.css` is the
  only stylesheet), but JavaScript-generated markup contains hundreds of inline `style="..."` attributes. New
  work should use classes in `styles.css`; migrating the existing ones is a long, safe, boring job for a rainy
  day.

## Suggested order of attack

1. **H1 (XSS)** — biggest real risk, and it only gets harder as more markup is added.
2. **M1 (books pending-sale)** — a visible correctness bug students will hit.
3. **M3, M5** — two-line robustness fixes; do them together.
4. **M4** — a real efficiency win, low risk.
5. **H2** — a decision, not a code change.
6. **L1, L2, L3** — dead-code cleanup, safe to batch.
